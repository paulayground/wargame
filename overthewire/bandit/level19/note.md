# 출제의도

`setuid`가 설정되어 있을 때의 위험성을 파악할 것

# 배운 점

`setuid`가 설정된 프로그램은 실행될 때 실행한 사용자 권한이 아니라 파일 소유자의 권한으로 실행되어 의도하지 않은 취약점이 노출될 수 있다.
`setuid`는 예를 들면 사용자가 비밀번호 변경을 하게되면 `/etc/shadow` 같은 파일을 수정해야하는데, 이 파일의 경우 권한이 root라서 `setuid` 없이는 root가 되거나 직접권한 받는 방법 말고는 수정할 수가 없다. 그래서 `setuid`가 설정된 파일들을 찾아 공격하기도 하며, `setuid` 취약점이 노출되지 않게 신경써야한다.

`env`가 환경변수를 확인하고 추가, 삭제하는 정도의 명령어 인줄 알았는데, env 통해 명령어를 실행시킬 수 있다는 것을 처음 알게됐다.(보통은 env로 환경변수를 설정하면서 프로그램을 실행하는 용도로 사용)

# 적용할 수 있는 곳

`find / -perm -4000 2>/dev/null`과 같이 `setuid`가 취약하게 설정된 것이 있는지 확인해봐야겠다.

# 사고 흐름 복기 & 후기

`setuid` 실행파일을 보고 바로 리버싱해야겠다는 생각이 들었으나, 먼저 실행을 해보자고 생각을 했고
실행파일에서 별도의 리버싱 없이 해결할 수 있는 힌트를 통해 클리어할 수 있었다.

# 다른 방법

`bandit20-do`을 리버싱해봤는데, 파라미터가 없을 때 힌트를 주며 종료되고, `/usr/bin/env`파라미터를 넘겨 실행하는 것으로 보임.
`bandit20-do`파일의 `setuid` 설정되어 있기때문에 실행할 때 Effective UID가 bandit20으로 바뀌고 해당 상태에서 env 명령어를 통해 응답을 가져오기 때문에 취약점이 발견된다는 것을 알 수 있다.
